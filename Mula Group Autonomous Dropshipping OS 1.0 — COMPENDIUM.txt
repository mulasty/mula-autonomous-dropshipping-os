# Mula Group Autonomous Dropshipping OS 1.0 — COMPENDIUM


Status: ACTIVE
Owner: Mula Group
Project mode: living knowledge base + implementation control center
Primary goal: build a modular autonomous dropshipping system operated by agent workflows with human control over exceptions.


---


## 1. Project vision


Mula Group Autonomous Dropshipping OS 1.0 is a production-oriented commerce operating system for automated online sales in a dropshipping model. The system is designed to import supplier data, normalize products, evaluate listing viability, generate listing content, publish offers, synchronize price and stock, route orders, support customers, and report exceptions.


The project is intentionally structured as a Codex-friendly repository with explicit architecture, modules, prompts, policies, logs, and implementation phases.


---


## 2. Core assumptions


1. BaseLinker is the operational commerce hub.
2. n8n is the workflow orchestration layer.
3. Supabase/Postgres is the source-of-truth data layer.
4. AI is the decision-support and generation layer, not the source of truth.
5. Every risky action must have rules, logs, and an escalation path.
6. Version 1.0 focuses on one supplier and one primary sales channel.


---


## 3. Strategic objective


Build an autonomous but controlled dropshipping system capable of handling 70–90% of repetitive operational work while escalating uncertain, legal, financial, or reputational exceptions to a human operator.

## 5. Operational modules


### Module A — Supplier Intake
Purpose: ingest supplier data into the system.


### Module B — Product Intelligence
Purpose: score product quality, margin, risk, and listing eligibility.


### Module C — Listing Factory
Purpose: generate titles, descriptions, attributes, SEO, and channel-ready payloads.


### Module D — Sync Engine
Purpose: keep prices and stock aligned with supplier changes and guardrails.


### Module E — Order Router
Purpose: validate and forward orders to supplier systems.


### Module F — Customer Support AI
Purpose: automate low-risk customer conversations and escalate sensitive cases.


### Module G — Control Tower
Purpose: central oversight, exceptions, KPIs, and implementation governance.

- pricing exceptions
- support automation rate
- escalation rate

---


## 8. Required governance rules


- no automated publication below minimum margin
- no automated publication for banned categories
- no automated response for complex complaints
- no silent price changes above defined threshold
- every AI action logged with timestamp and context
- every failed workflow added to exception queue
